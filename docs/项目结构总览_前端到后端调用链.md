# 项目结构总览：前端页面、前端脚本与后端服务调用链

> 目标：让你第一次接手这个仓库时，先快速看懂“哪些目录值得看、页面和脚本分别干什么、前端到底打到了哪些后端服务、后端内部又是怎么分层的”。

---

## 一、推荐阅读顺序

如果你现在是第一次接手这个项目，建议按下面顺序阅读：

1. `README.md`
   - 看仓库的整体定位：这是一个“Python 学习 + 前后端联动演示”的综合仓库。
2. `frontend/README.md`
   - 看前端页面清单、端口约定、mock 模式和页面与服务的对应关系。
3. 本文
   - 看“目录职责 + 页面到后端的真实调用链 + 各服务内部结构”。
4. 各服务自己的 `09_项目说明.md`
   - 当你准备深入某个服务时，再去看对应服务的说明文档。

## 二、仓库顶层目录做什么

### 1. 学习内容目录

- `01.Python语言基础/`
  - Python 基础章节，偏“语言学习材料 + demo + practice”。
- `02.开发环境及框架介绍/`
  - 框架专题与实验材料，`FastAPI 深度专题` 课程内容也在这里。
- `03.项目实战/`
  - 独立项目练习源码与说明文档。

这三块更偏“课程与练习素材”。

### 2. 前端目录

- `frontend/`
  - 当前最需要接手的静态前端页面目录。
  - 这里没有构建工具，主要是 **HTML + CSS + 原生 JavaScript**。
  - 页面既承担“课程导航/文档浏览”，也承担“登录 / 管理后台 / 日志侦探”等前后端联调演示。

### 3. 后端服务目录

- `rbac_auth_service/`
  - 用户、角色、权限、Todo、Project、Task 的主服务。
  - 也是 `login.html`、`admin.html` 的主要上游。
- `integration_gateway_service/`
  - 网关服务。
  - 负责 JWT 验证、请求转发、统一响应包装、部分审计日志上报。
- `backend_user_order_service/`
  - 被网关代理调用的下游示例业务服务。
- `log_audit_service/`
  - 审计日志服务。
  - `admin.html` 的日志面板会直接读取它。
- `log_detective_service/`
  - 日志分析服务。
  - `log-detective.html` 不直接打它，而是通过网关转发过去。

### 4. 文档与测试

- `docs/`
  - 综合导航、架构文档、快速开始、学习路线。
- `tests/`
  - 根级测试资源。
- 各服务目录下也各自带有测试文件。

## 三、前端目录怎么读

## 1. 页面类 HTML

- `portal.html`
  - 前端入口总导航。
  - 更像“总菜单 / 总跳板页”，不直接承担复杂业务。
- `index.html`
  - Python 学习导航主页。
  - 左侧章节导航，右侧文档与在线 Python 执行器。
- `projects.html`
  - 项目实战导航页。
  - 结构和 `index.html` 类似，但内容换成项目实战索引。
- `doc-viewer.html`
  - Markdown 文档浏览器。
  - 通过 URL 参数 `file=...` 动态加载仓库内文档。
- `fastapi-course/index.html`
  - FastAPI 深度专题的课程导航页。
  - 左侧课程目录，右侧 Markdown 预览。
- `login.html`
  - RBAC 登录页。
- `admin.html`
  - RBAC 管理后台。
- `log-detective.html`
  - 日志侦探演示页。

## 2. 前端公共脚本

- `config.js`
  - 所有页面共用的“后端地址配置中心”。
  - 前端真正打到哪个端口，先看这里。
- `common.js`
  - 公共工具脚本。
  - 提供：
    - `apiClient()`：统一请求封装
    - `renderGenericTable()`：通用表格渲染
    - 调试面板日志输出
    - Toast 提示
- `mock-data.js`
  - mock 数据字典。
  - 主要给 `login.html`、`admin.html` 做离线演示。
- `mock-api.js`
  - fetch 拦截器。
  - mock 模式打开后，会优先返回 `mock-data.js` 中的数据。
- `fastapi-course/js/navigation.js`
  - FastAPI 深度专题页面的导航、Markdown 加载、进度条、主题切换逻辑。

## 3. 一个重要现状

这个前端目录里，**很多页面不是“HTML + 独立 JS 模块”分离结构，而是“HTML + 内联 script”结构**：

- `portal.html`
- `index.html`
- `projects.html`
- `login.html`
- `admin.html`
- `log-detective.html`
- `doc-viewer.html`

因此排查时通常要同时看：

1. 页面结构区块
2. 页面底部内联脚本
3. 共享脚本（`config.js` / `common.js` / `mock-api.js`）

## 四、后端目录怎么读

### 1. `rbac_auth_service/`：结构最完整的主服务

这是仓库里最接近“标准分层后端”的服务。

核心分层如下：

- `app/main.py`
  - FastAPI 应用入口
  - 创建表
  - 挂载路由
  - 注册统一异常处理
- `app/config.py`
  - 数据库、Redis、JWT、审计日志地址等配置
- `app/database.py`
  - SQLAlchemy 引擎、Session、`get_db()`
- `app/models.py`
  - ORM 实体模型
- `app/schemas.py`
  - 请求/响应 DTO、统一响应模型
- `app/security.py`
  - 密码哈希、JWT、黑名单
- `app/dependencies.py`
  - 当前用户、权限检查、数据库依赖
- `app/routers/`
  - API 路由层
- `app/services/`
  - 业务编排层
- `app/repositories/`
  - 持久化读写层
- `init_rbac_data.py`
  - 初始化 admin、角色、权限等数据

它的真实主链是：

`router -> service -> repository -> model/database`

### 2. `integration_gateway_service/`：网关层

结构比 RBAC 轻，但职责很清晰：

- `app/main.py`
  - 入口
- `app/config.py`
  - 网关配置
- `app/dependencies.py`
  - JWT 解析、客户端依赖注入
- `app/client_backend_service.py`
  - 调后端用户/订单服务
- `app/log_detective_client.py`
  - 调日志侦探服务
- `app/log_audit_client.py`
  - 上报审计日志
- `app/routers/gateway.py`
  - 真正对外暴露的网关接口

它的真实主链是：

`前端/调用方 -> gateway router -> client -> 下游服务`

### 3. `backend_user_order_service/`：下游示例业务服务

这是一个很轻量的下游服务：

- `app/main.py`
  - 直接定义用户查询与订单创建接口
- `app/schemas.py`
  - 请求/响应模型

这里没有再拆 service/repository，主要是教学示例。

### 4. `log_audit_service/`：审计日志服务

结构类似简化版 CRUD 服务：

- `app/main.py`
  - 入口 + 建表
- `app/config.py`
  - 数据库配置
- `app/database.py`
  - 数据库连接与 Session
- `app/models.py`
  - 审计日志模型
- `app/repositories/audit_log_repository.py`
  - 审计日志增查逻辑
- `app/routers/audit_logs.py`
  - 对外接口 + 简单 HTML 查看页
- `app/schemas.py`
  - 输入输出模型

### 5. `log_detective_service/`：日志分析服务

核心关注点不是持久化，而是“分析逻辑”：

- `app/main.py`
  - 入口
- `app/config.py`
  - 分析限制配置
- `app/analyzer.py`
  - 日志解析与统计的核心算法
- `app/routers/log_detective.py`
  - 对外分析接口
- `app/schemas.py`
  - 输入输出模型

## 五、页面、脚本、后端之间的真实关系

## 1. `portal.html`

职责：

- 汇总所有入口
- 不负责复杂业务处理

关系：

- 读取 `config.js` 显示当前 API 地址
- 主要跳转到其他页面或文档查看器

调用链：

`portal.html -> config.js -> 只显示地址 / 跳转其他页面`

## 2. `doc-viewer.html`

职责：

- 根据 URL 里的 `file` 参数动态拉取 Markdown
- 在页面中渲染文档

关系：

- 使用 `marked` 做 Markdown 转 HTML
- 使用 `highlight.js` 做代码高亮

调用链：

`doc-viewer.html -> fetch(仓库内 Markdown 文件) -> marked/highlight.js -> 渲染结果区`

注意：

- 这是“前端读静态文档”，不是调用后端业务 API。

## 3. `index.html`

职责：

- Python 章节导航
- 文档展示
- 在线代码编辑与运行

关系：

- 页面内联脚本保存 `chapters` 配置
- 页面自己负责：
  - 导航渲染
  - Markdown 加载
  - 代码文件加载
  - `input()` 到浏览器模式的适配
  - Pyodide 初始化与执行

调用链：

`index.html -> 内联 script -> fetch(章节 Markdown / demo 代码) -> 右侧文档区 + 在线执行器`

注意：

- 这个页面主要面向本仓库静态内容，不直接打后端微服务 API。

## 4. `projects.html`

职责：

- 项目导航
- 项目说明展示
- 示例代码预览与运行

关系：

- 结构和 `index.html` 非常像
- 只是数据源从“章节”换成了“项目列表”

调用链：

`projects.html -> 内联 script -> fetch(项目说明 Markdown / 示例代码) -> 文档区 + 在线执行器`

## 5. `fastapi-course/index.html`

职责：

- FastAPI 深度专题课程导航
- 章节 / lab 的 Markdown 预览

关系：

- 页面结构在 HTML
- 交互逻辑主要在 `fastapi-course/js/navigation.js`

调用链：

`fastapi-course/index.html -> navigation.js -> fetch(课程 Markdown / lab README) -> 右侧预览区`

## 6. `login.html`

职责：

- 登录并保存 JWT
- 按当前用户身份跳转

关系：

- 依赖：
  - `mock-data.js`
  - `config.js`
  - `mock-api.js`
- 页面底部内联 script 负责：
  - 主题切换
  - 表单提交
  - 请求 `/auth/login`
  - 成功后再请求 `/auth/me`

调用链：

`login.html -> 内联 script -> POST /auth/login -> rbac_auth_service`

成功后补一跳：

`login.html -> GET /auth/me -> rbac_auth_service -> 判断是否超级管理员 -> 跳 admin.html 或 index.html`

## 7. `admin.html`

职责：

- RBAC 管理后台
- 用户 / 角色 / 权限管理
- 审计日志查看

关系：

- 依赖：
  - `mock-data.js`
  - `config.js`
  - `mock-api.js`
  - `common.js`
- 页面底部内联 script 负责：
  - token 校验
  - 面板切换
  - 加载 dashboard
  - 加载 users / roles / permissions
  - 创建用户 / 角色
  - 读审计日志
  - 调试面板开关

主要调用链：

1. 用户信息校验  
   `admin.html -> GET /auth/me -> rbac_auth_service`

2. 看板统计  
   `admin.html -> GET /users/ / GET /rbac/roles / GET /rbac/permissions -> rbac_auth_service`

3. 用户管理  
   `admin.html -> GET/PATCH/POST /users/ -> rbac_auth_service`

4. 角色管理  
   `admin.html -> GET/POST /rbac/roles -> rbac_auth_service`

5. 权限查看  
   `admin.html -> GET /rbac/permissions -> rbac_auth_service`

6. 审计日志  
   `admin.html -> GET /logs -> log_audit_service`

## 8. `log-detective.html`

职责：

- 输入日志文本
- 调用网关进行分析
- 展示统计、可疑 IP、关键错误

关系：

- 依赖：
  - `config.js`
  - `common.js`
- 页面内联 script 负责：
  - 加载示例日志
  - 提交分析请求
  - 渲染统计卡片与两个表格

真实调用链：

`log-detective.html -> apiClient() -> POST /gateway/log-detective/analyze -> integration_gateway_service`

接着是：

`integration_gateway_service -> log_detective_client -> POST /internal/log-detective/analyze -> log_detective_service`

## 六、最重要的后端调用链

## 1. 登录链路

```text
login.html
  -> POST /auth/login
  -> rbac_auth_service/app/routers/auth.py
  -> security.py / dependencies.py / models.py
```

补充一跳：

```text
login.html
  -> GET /auth/me
  -> rbac_auth_service/app/routers/auth.py
  -> dependencies.py 获取当前用户
```

## 2. 管理后台链路

```text
admin.html
  -> 内联 script 中的 apiFetch()
  -> rbac_auth_service/app/routers/users.py / roles.py / auth.py
  -> app/services/*
  -> app/repositories/*
  -> app/models.py + app/database.py
```

日志面板则是：

```text
admin.html
  -> GET /logs
  -> log_audit_service/app/routers/audit_logs.py
  -> repositories/audit_log_repository.py
  -> models.py / database.py
```

## 3. 日志侦探链路

```text
log-detective.html
  -> common.js: apiClient()
  -> integration_gateway_service/app/routers/gateway.py
  -> integration_gateway_service/app/log_detective_client.py
  -> log_detective_service/app/routers/log_detective.py
  -> log_detective_service/app/analyzer.py
```

## 4. 网关代理业务链路

```text
调用方
  -> integration_gateway_service/app/routers/gateway.py
  -> client_backend_service.py
  -> backend_user_order_service/app/main.py
```

创建订单时还会附带：

```text
gateway.py
  -> log_audit_client.py
  -> log_audit_service /logs
```

## 八、当前项目的几个接手结论

1. 这是一个 **静态前端 + 多个 FastAPI 服务** 的教学仓库。
2. 前端并不是完全模块化的工程化前端，很多页面仍是 **HTML + 内联脚本**。
3. 后端里：
   - `rbac_auth_service` 分层最完整；
   - `integration_gateway_service` 负责串起多个下游服务；
   - `log_detective_service` 的核心价值主要在 `analyzer.py`；
   - `backend_user_order_service` 是最轻量的下游演示服务。
4. 真正最值得先看懂的三条链路是：
   - `login.html -> rbac_auth_service`
   - `admin.html -> rbac_auth_service + log_audit_service`
   - `log-detective.html -> gateway -> log_detective_service`

如果你接下来要继续改注释、排查流程或做结构优化，这三条链路就是最优先入口。
